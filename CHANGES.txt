========================================
DATABASE SCHEMA ALIGNMENT:
========================================

The changes align with the actual doctors table schema which includes:
- doctor_id (PRIMARY KEY, AUTO_INCREMENT)
- staff_id (FOREIGN KEY, UNIQUE)
- specialization (VARCHAR 100, NOT NULL)
- consultation_fee (DECIMAL 10,2, NOT NULL)
- max_appointments_per_day (INT, DEFAULT 10)
- created_at (TIMESTAMP)

Note: The following fields do NOT exist in the doctors table:
- available_days (handled separately in doctor_availability table)
- bio (not in current schema)


========================================
WORKFLOW FLOW:
========================================

CREATE NEW DOCTOR:
1. Admin fills form with doctor fields
2. Frontend sends POST with staff_id = null
3. API receives request, checks staff_id
4. staff_id is null/missing, so calls createStaff()
5. createStaff() creates:
   - users table entry
   - staff table entry
   - doctors table entry with specialization and consultation_fee

EDIT DOCTOR (NO ROLE/DEPT CHANGE):
1. Admin loads existing doctor, modifies fields
2. Frontend sends POST with staff_id and existing data
3. API receives request, checks staff_id
4. Validates old role vs new role (same role = no change)
5. Validates old dept vs new dept (same dept = no change)
6. Both same, so calls updateStaff()
7. updateStaff() updates:
   - users table (email)
   - staff table (name, designation, hire_date, etc.)
   - doctors table (specialization, consultation_fee)
8. Returns 200 status (update success)

EDIT DOCTOR (WITH ROLE/DEPT CHANGE):
1. Admin changes doctor's role or department
2. Frontend sends POST with staff_id and new role/dept
3. API receives request, checks staff_id
4. Validates old role vs new role (DIFFERENT = change detected)
5. OR validates old dept vs new dept (DIFFERENT = change detected)
6. Shows confirmation warning to user
7. If confirmed, calls recreateStaffWithRoleChange()
8. Recreates entire staff record with new role/dept
9. All appointments/encounters set to NULL (orphaned)
10. Returns 201 status (new record created)


========================================
TRANSACTION SAFETY:
========================================

All operations now use MySQL transactions (BEGIN/COMMIT/ROLLBACK):

- createStaff(): Uses transaction to ensure users + staff + doctors all created together
- updateStaff(): Uses transaction to ensure users + staff + doctors all updated together
- recreateStaffWithRoleChange(): Uses transaction to ensure old deleted + new created atomically

This prevents partial updates and maintains data integrity.

4. DOCTOR APPOINTMENT DEPARTMENT FIX
========================================

ISSUE FIXED:
When doctor tried to move an appointment to their department using "Move to My Department" button, 
the API returned: "Unknown column 'a.appointment_id' in 'field list'" error.

ROOT CAUSE:
In services/doctor.js, the updateAppointmentDepartment() function had a SELECT query that used 
table alias 'a' but didn't define it in the FROM clause:
  ❌ BROKEN: SELECT a.appointment_id, a.doctor_id FROM appointments 
  ✅ FIXED:  SELECT a.appointment_id, a.doctor_id FROM appointments a

FIX APPLIED:
File: services/doctor.js (updateAppointmentDepartment function)
- Line: Added table alias 'a' to the FROM clause
- Changed: FROM appointments 
- To:      FROM appointments a

RESULT:
- Doctor can now successfully move appointments to their department
- Appointment department_id is updated to match doctor's department_id
- Button "Move to My Department" now works without errors
- Resolves the department mismatch warning when viewing appointments
