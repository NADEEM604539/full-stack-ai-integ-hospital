SECTION 2: AI INTEGRATION - DOCTOR TREATMENT ASSISTANT
================================================================================

AI FEATURE: Comprehensive Treatment Assistant for Doctors

🏥 WHAT THE AI DOES:

1️⃣  **SOAP NOTE STRUCTURING**
├─ Doctor records clinical notes (voice or text)
├─ AI organizes into SOAP format
├─ Doctor reviews and approves
└─ Improves documentation speed & quality

2️⃣  **DIAGNOSIS SUGGESTIONS** (AI-Assisted, Doctor Decides)
├─ AI analyzes subjective + objective findings
├─ Generates differential diagnosis list (ranked by probability)
├─ Doctor reviews and selects/customizes final diagnosis
├─ Final diagnosis saved by DOCTOR DECISION (not AI)
└─ Uses patient history + medical guidelines

3️⃣  **MEDICATION RECOMMENDATIONS** (AI-Assisted, Doctor Decides)
├─ AI analyzes:
│  ├─ Patient diagnosis
│  ├─ Allergies and contraindications
│  ├─ Current medications (drug-drug interactions)
│  └─ Patient vital signs & conditions
├─ AI suggests medications with dosages & evidence
├─ Shows contraindications and warnings
├─ Doctor reviews and approves/customizes
└─ Final prescription written by DOCTOR

## HOW IT WORKS (STEP-BY-STEP)

### **Step 1: Doctor Inputs Clinical Data**
Doctor submits unstructured notes:
```
"45M chest pain 2 days, worse with exertion, sweating. 
BP 140/90, HR 95, EKG normal, Troponin negative.
PMH: Hypertension on lisinopril. 
Allergy: PCN"
```

### **Step 2: AI Structures & Analyzes**
AI returns structured response:
```json
{
  "soap_note": {
    "subjective": "45M with exertional chest pain x2 days...",
    "objective": "BP 140/90, HR 95, EKG normal, Troponin negative...",
    "assessment": "Requires diagnosis review",
    "plan": "Recommendations below"
  },
  
  "differential_diagnoses": [
    {"diagnosis": "Atypical angina", "probability": "45%", "reasoning": "Exertional, sweating"},
    {"diagnosis": "Musculoskeletal pain", "probability": "35%", "reasoning": "But vital signs elevated"},
    {"diagnosis": "GERD", "probability": "15%", "reasoning": "Less likely with exercise trigger"},
    {"diagnosis": "Anxiety", "probability": "5%", "reasoning": "Normal EKG & troponin"}
  ],
  
  "medication_recommendations": [
    {
      "drug": "Aspirin", 
      "dose": "325mg PO daily",
      "indication": "Chest pain, stable vitals",
      "contraindications": "None detected",
      "interactions": "Safe with lisinopril"
    },
    {
      "drug": "Atenolol",
      "dose": "25mg PO daily",
      "indication": "Hypertension management, cardiac protection",
      "contraindications": "Monitor for bradycardia",
      "interactions": "None with current meds"
    }
  ],
  
  "warnings": ["Monitor troponin trend", "Consider stress test if symptoms persist"],
  "completeness": "95%"
}
```

### **Step 3: Doctor Reviews & Decides**
✓ Reviews AI suggestions
✓ Selects diagnosis from differential
✓ Approves or modifies medications
✓ Makes final clinical decisions
✓ Documents decision

### **Step 4: Save to Database**
✓ SOAP note saved with AI confidence scores
✓ Doctor's final diagnosis recorded
✓ Audit trail: AI suggestion vs Doctor decision
✓ Enables learning & compliance

SECTION 3: CLERK AUTHENTICATION & MYSQL INTEGRATION
================================================================================

AUTHENTICATION FLOW:
1. User logs in via Clerk UI
2. Clerk returns user_id and session token
3. Application stores clerk_user_id in MySQL users table
4. Backend queries users table to get role_id
5. Middleware loads permissions from role_permissions table
6. All subsequent requests use cached permissions

CLERK ↔ MYSQL SYNC:
├─ users.clerk_user_id = VARCHAR(255) NOT NULL UNIQUE (Clerk's external ID)
├─ users.role_id = INT (links to roles table)
├─ staff.user_id = INT (links to users table for hospital staff)
└─ patients.user_id = INT (links to users table for patient portal)

PERMISSION LAYERS:

Layer 1 - AUTHENTICATION (Clerk)
├─ User logs in via Clerk UI
└─ Session token stored in browser & backend

Layer 2 - AUTHORIZATION (Next.js Middleware)
├─ Verify Clerk session
├─ Load user role from MySQL users table
├─ Load permissions from role_permissions table
├─ Block request if user lacks permission (403 Forbidden)
└─ ⚠️ CRITICAL: Authorization BEFORE database query

Layer 3 - DATABASE SECURITY
├─ clerk_user_id synced with Clerk (unique constraint)
├─ created_by & updated_by fields track changes
├─ is_deleted flag for soft deletes (not hard delete)
├─ deleted_at timestamp for recovery
├─ audit_logs table for compliance
└─ Row-level filtering in backend queries (not DB views)

SECTION 4: TRANSACTION BOUNDARIES (Backend Implementation)
================================================================================

ALL complex operations are ACID transactions in BACKEND:

BOOKING APPOINTMENT:
1. Check doctor availability (SELECT)
2. Validate time slot not taken
3. INSERT appointment record
4. COMMIT on success, ROLLBACK on conflict

DISPENSING MEDICATION:
1. Load prescription details
2. Check inventory levels (inventory_items.quantity_in_stock)
3. UPDATE inventory quantity
4. INSERT inventory_transactions record
5. UPDATE prescription status to 'Dispensed'
6. COMMIT

CREATING INVOICE:
1. Load encounter services (consultation, meds, lab, bed)
2. Calculate subtotal, tax, discount, total
3. INSERT invoice record
4. INSERT invoice_line_items records
5. COMMIT

ADMITTING PATIENT:
1. Create encounter record
2. Link to patient and doctor
3. Record admission_date = NOW()
4. Set status = 'Active'
5. COMMIT

SECTION 5: SOFT DELETE STRATEGY (Data Safety & Compliance)
================================================================================

SOFT DELETE FIELDS on critical tables:
├─ is_deleted BOOLEAN DEFAULT FALSE
└─ deleted_at TIMESTAMP NULL

DELETION FLOW:
1. UPDATE record SET is_deleted = TRUE, deleted_at = NOW()
2. Backend logs change to audit_logs table
3. Record remains in database but excluded from queries

RECOVERY (Admin Only):
1. Query with is_deleted = TRUE to view deleted records
2. Update is_deleted = FALSE, deleted_at = NULL to restore
3. Keep records permanently for audit trails

APPLIES TO: patients, appointments, encounters, prescriptions, invoices

SECTION 6: CORE BUSINESS RULES
================================================================================

APPOINTMENTS:
✓ Prevent double-booking (TRG_T1 enforces strict interval overlap detection)
✓ Validate doctor availability (shift times from doctor_availability table)
✓ Enforce doctor daily appointment limit (max_appointments_per_day from doctors table)
✓ Dynamic 30-minute default duration with custom support
✓ Doctor availability checked in trigger + function FN4
✓ Soft delete when cancelled (is_deleted = TRUE)

INVENTORY:
✓ Prevent stock-outs (sp_dispense_medication uses FOR UPDATE row locking)
✓ Race condition detection (TRG_T3 validates stock before deducting)
✓ Auto-deduct on dispensing (trigger + procedure safeguards)
✓ Trigger prevents negative inventory (CHECK constraint + validation)
✓ Reorder alerts via vw_low_stock_items view

BILLING:
✓ Calculate totals in backend (with TRG_T9 integrity validation)
✓ Components: Consultation + Meds + Lab + Bed charges
✓ Tax applied: tax_percent * subtotal
✓ Payment status: Paid/Partial/Unpaid
✓ TRG_T9 prevents manual corruption (total = subtotal + tax - discount)

ENCOUNTERS:
✓ Linked to appointment
✓ One SOAP note per encounter (UNIQUE constraint)
✓ Soft delete on discharge (is_deleted = TRUE)
✓ created_by tracks who created encounter

AUDIT & COMPLIANCE:
✓ Every record has created_by & updated_by
✓ Soft deletes enable recovery
✓ Audit logging via backend (manual inserts)
✓ Clerk integration enables authentication audit

## 🔐 DEPARTMENT ACCESS RULES

### RECEPTIONIST (Role=4, Admin Department)
```
Can Register Patients To Departments:
├─ Assigns new patients to specific department
├─ department_id field in patients table
├─ Links patient to department-specific care pathway
└─ Example: Patient registered to "Cardiology" (dept=2)

Can Book Appointments In:
├─ ANY clinical department (1-6)
├─ VALIDATES patient-department alignment before booking
├─ Validates doctor-department alignment
├─ Sees all available slots across departments
└─ Books globally but slots are department-specific

APPOINTMENT VALIDATION:
├─ Step 1: Verify patient.department_id matches requested department
├─ Step 2: Verify doctor.department_id matches requested department
├─ Step 3: Verify patient.department_id == doctor.department_id
├─ FAIL if any mismatch (400 error: department mismatch)
└─ ONLY proceed if all three are consistent

Cannot Access:
├─ Clinical notes from other departments
├─ Doctor schedules for other non-appointment purposes
└─ Clinical decision-making
```

### DOCTOR (Role=2, e.g., General Medicine)
```
Restricted To Own Department:
├─ View only own appointments (department=1)
├─ View only own patients (referral to dept=1)
├─ Record vitals for own department
├─ Write prescriptions for own cases
└─ Cannot view Cardiology patient data

Access Model:
SELECT * FROM appointments WHERE doctor_id=X AND department_id=1
↑ department_id must match staff.department_id
```

### NURSE (Role=3, e.g., Cardiology)
```
Department-Based Patient Access:
├─ View ONLY patients registered to own department
├─ Query: SELECT * FROM patients WHERE department_id = (SELECT department_id FROM staff WHERE user_id = X)
├─ Example: Nurse in Cardiology (dept=2) sees ONLY dept=2 patients
├─ Cannot view General Medicine patients even with SELECT privilege
└─ Department isolation enforced at both DB and backend

Restricted To Own Department:
├─ View only assigned patients in own department
├─ Record vitals only for own department patients
├─ See only appointments for own department
└─ Cannot access other department data

Access Model:
SELECT * FROM vitals WHERE recorded_by=X AND department_id=2
  ↑ department_id must match staff.department_id ONLY
```

### PHARMACIST (Role=5, Pharmacy Department)
```
Cross-Department Access:
├─ Views prescriptions from ALL departments
├─ Dispenses medications for ALL departments
├─ Manages inventory for hospital-wide use
└─ Records pharmacy transactions per-department

Access Model:
SELECT * FROM prescriptions ← No department filter
  + Tracks destination_department for each prescription
```

### FINANCE (Role=6, Administration Department)
```
Cross-Department Access:
├─ Views invoices from ALL departments
├─ Processes payments for ALL departments
├─ Generates reports by-department
└─ Reconciles billing per-department

SELECT SUM(amount) FROM billing GROUP BY department_id
  + Can filter by department for reconciliation
```

```
Full Hospital Access:
├─ Oversees ALL departments
├─ Manages staff assignments per-department
├─ Approves doctor availability changes
├─ Views all audit logs and analytics
└─ System configuration

Access Model:
SELECT * ← No department filtering, admin sees all
```

```
Personal Health Records - Department-Aware:
├─ View only OWN medical records (row-level filtering)
├─ Registered to ONE department (patients.department_id)
├─ Access all own appointments in registered department
├─ View own prescriptions from registered department
├─ View own billing and payment history
└─ READ-ONLY access (cannot modify any records)

Department Registration Workflow:
├─ Receptionist registers patient to department during intake
├─ patient.department_id assigned at registration
├─ Example: Patient John Doe registered to Cardiology (dept=2)
├─ Patient can only book appointments in that department
└─ All care records are department-specific

Example Queries for Patient John Doe (patient_id=5, dept=2):

│  SELECT * FROM appointments WHERE patient_id=5 AND department_id=2
├─ View own vitals:
│  SELECT * FROM vitals WHERE patient_id=5 AND department_id=2
├─ View own prescriptions:
│  SELECT * FROM prescriptions WHERE patient_id=5 AND department_id=2
└─ Cannot view:
   ├─ Other patients' data
   ├─ Appointments in other departments
   └─ Administrative or staff data
```
